# Architecture

The pipeline turns Taiwan TGOS address CSVs, the MOI administrative
boundary shapefiles, and an OpenStreetMap PBF into a set of SQLite
databases that an ATAK plugin can read offline.

## Three source streams

1. **TGOS CSVs** (input/) — government-issued, per-county address points
   with exact coordinates. Taichung is published with both TWD97 (EPSG
   3826) and WGS84 (EPSG 4326); Changhua publishes only TWD97 and we
   reproject to WGS84 via `pyproj`. Each row carries the MOI 7- or
   8-digit district code, which we map to county/district names via
   `config/moi_district_codes.yaml`. The mapping was generated by
   polygon-in resolution against admin polygons, not hand-written.

2. **MOI boundary shapefiles** (input/moi-boundaries/) — the Ministry of
   the Interior's authoritative 直轄市/縣市界線 (`COUNTY_MOI`) and
   鄉鎮市區界線 (`TOWN_MOI`) shapefiles, release 1140318. `extract_
   townships.py` reads these with `pyshp` to build `townships.sqlite`
   (admin_level 4/7/8 polygons). This is the legal ground truth — it
   replaced the earlier OSM-derived townships layer. Each 鄉鎮市區 polygon
   carries its parent 縣市 inline, so county membership is a column read,
   not a runtime spatial join. Shapefiles are GCS_TWD97[2020]; the datum
   offset from WGS84 is sub-metre, so coordinates are stored verbatim as
   WGS84 lon/lat. Like the TGOS CSVs, these are large source data mounted
   read-only and are git-ignored — place them under `input/moi-boundaries/`
   before running.

3. **OSM PBF** (cache/) — Geofabrik's Taiwan extract. Clipped to the
   region bbox with `osmium extract --bbox` to produce a ~70 MiB
   `tw-central.osm.pbf`. Supplies the **roads** layer (`extract_roads.py`)
   and the non-TGOS-county **landmark/address** layer
   (`extract_places_osm.py`); it no longer drives the townships layer.

## Why per-county SQLite

The plan's design choice (matches what `glob places-*.sqlite` expects on
the plugin side):

- TGOS releases each county independently — they should ship
  independently.
- Plugin can mount only the counties it cares about.
- Re-ingest cost for one county is ~10 min worst case (Taichung); the
  others stay untouched.

## Schema reuse

`places-taichung.sqlite`, `places-changhua.sqlite` and `places-osm.sqlite`
share an **identical** schema (modulo the `source` column carrying
`'tgos'` or `'osm'`). The plugin opens all three with the same code
path; the `source` column lets the UI annotate row provenance
(authoritative vs. crowdsourced).

`townships.sqlite` and `roads.sqlite` are stand-alone: their R*Tree
spatial indexes drive offline reverse geocoding.

Since schema version 2 (see [`data-contract.md`](./data-contract.md)),
the `places-*.sqlite` files also carry a `places_rtree` spatial index so
plugins can do nearest-address reverse geocoding in < 200 ms against
~731K rows. The full plugin-facing schema, naming, and query patterns are
all documented in [`data-contract.md`](./data-contract.md); when in
doubt, that file is the canonical source of truth.

## Name normalisation

Both OSM and the MOI shapefiles use the older glyph 「臺」 (in 臺中市 /
臺北市 / 臺南市 / 臺東縣 / 臺西鄉…), while TGOS uses 「台」 throughout.
Every extract script (`extract_townships.py`, `extract_roads.py`,
`extract_places_osm.py`) applies the same one-character translation table
so all stored names use 「台」. This is not cosmetic: county names are
join keys. `extract_places_osm.py` excludes TGOS-covered counties with
`WHERE name_zh IN ('台中市','彰化縣')`, and `reverse_geocode.py` routes
Tier 3 by `{'台中市': 'places-taichung.sqlite', …}` — if the MOI layer
stored 「臺中市」 both would silently miss, duplicating addresses and
scanning the wrong database.

Fullwidth digits in TGOS addresses (`２之３之２號`) are preserved in
`display_name` for human-facing UI but mirrored as halfwidth + hyphenated
in `display_name_halfwidth` (`2-3-2號`) for FTS5 search.

## Coordinate transform

`pyproj` is configured with EPSG:3826 (TWD97 / TM2 zone 121) ↔ EPSG:4326
(WGS84). Round-trip stability is verified at 0 m on every dev environment
via `scripts/coord_transform.py` self-test, and ground-truth landmark
checks come from the TGOS Taichung CSV (which carries both coordinate
systems for every row — the data itself is the projection's reference).

## Address reduction (two stages)

Once per-county ingest is done, two reduction passes run by default:

1. `scripts/dedup_floors.py` — derives a base number by removing a
   `樓`/`層` suffix only when it occurs after the first `號`, then groups by
   exact coordinate and complete base address. An explicit base row wins;
   when a group contains only floor rows, the lowest-id row is rewritten
   to the base number.
2. `scripts/collapse_coords.py` — removes only remaining duplicate rows
   with the same exact coordinate and complete base-address key.

The end result permits multiple different addresses at one TGOS coordinate,
but guarantees at most one row per **coordinate plus complete base
address**. Strategy details, metadata layout, and current counts are in
[`dedup-floors-report.md`](./dedup-floors-report.md).

The verifier (`verify_samples.py`) skips CSV rows whose `number`
contains 樓/層 so its sampling stays aligned with what survives
stage 1.

## Verification

The verifier (`verify_samples.py`) is the single most important quality
gate in the pipeline. For each TGOS county we draw 50 anchor samples
from each of the four cardinal extremes (200 per county) and run six
checks per sample, totalling 2,400 assertions. See the manual for the
check list and tolerance budget.

## Output packaging

`build_manifest.py` ZIPs the sqlite files into four kits:

- `base.zip` — townships + roads + places-osm (must-have)
- `places-<county>.zip` — per-county TGOS sqlite (optional)
- `tw-central-full.zip` — everything in one bundle (convenience)

Each ZIP gets a sidecar `*.manifest.txt` carrying ZIP SHA-256, per-file
SHA-256, source CSV SHA-256, TGOS data date, OSM extraction counts, ISO
build timestamp, region bbox, release `data_version`, and
`address_policy_version`. `config/data_version.yaml` is the single source
of truth. Packaging refuses SQLite artifacts whose embedded version is
stale, and `scripts/check_data_version.py` verifies metadata, ZIP version
sidecars, and manifest hashes before release.
