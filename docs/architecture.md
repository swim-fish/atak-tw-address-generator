# Architecture

The pipeline turns Taiwan TGOS address CSVs plus an OpenStreetMap PBF
into a set of SQLite databases that an ATAK plugin can read offline.

## Two source streams

1. **TGOS CSVs** (input/) — government-issued, per-county address points
   with exact coordinates. Taichung is published with both TWD97 (EPSG
   3826) and WGS84 (EPSG 4326); Changhua publishes only TWD97 and we
   reproject to WGS84 via `pyproj`. Each row carries the MOI 7- or
   8-digit district code, which we map to county/district names via
   `config/moi_district_codes.yaml`. The mapping was generated by
   polygon-in resolution against OSM admin polygons, not hand-written.

2. **OSM PBF** (cache/) — Geofabrik's Taiwan extract. Clipped to the
   region bbox with `osmium extract --bbox` to produce a ~70 MiB
   `tw-central.osm.pbf`. Admin polygons for county-level (admin_level=4)
   span the bbox and cannot be reassembled from the clip, so `extract_
   townships.py` reads level=4 from the full PBF and level=7/8 from the
   clipped one.

## Why per-county SQLite

The plan's design choice (matches what `glob places-*.sqlite` expects on
the plugin side):

- TGOS releases each county independently — they should ship
  independently.
- Plugin can mount only the counties it cares about.
- Re-ingest cost for one county is ~10 min worst case (Taichung); the
  others stay untouched.

## Schema reuse

`places-taichung.sqlite`, `places-changhua.sqlite` and `places-osm.sqlite`
share an **identical** schema (modulo the `source` column carrying
`'tgos'` or `'osm'`). The plugin opens all three with the same code
path; the `source` column lets the UI annotate row provenance
(authoritative vs. crowdsourced).

`townships.sqlite` and `roads.sqlite` are stand-alone: their R*Tree
spatial indexes drive offline reverse geocoding.

## Name normalisation

OSM frequently uses the older glyph 「臺」 (in 臺中市 / 臺北市 / 臺南市
/ 臺東縣 / 臺西鄉…), while TGOS / MOI use 「台」 throughout. The
extract scripts apply a one-character translation table so all stored
names use 「台」, ensuring consistency across reverse-lookup outputs and
TGOS-vs-OSM join keys.

Fullwidth digits in TGOS addresses (`２之３之２號`) are preserved in
`display_name` for human-facing UI but mirrored as halfwidth + hyphenated
in `display_name_halfwidth` (`2-3-2號`) for FTS5 search.

## Coordinate transform

`pyproj` is configured with EPSG:3826 (TWD97 / TM2 zone 121) ↔ EPSG:4326
(WGS84). Round-trip stability is verified at 0 m on every dev environment
via `scripts/coord_transform.py` self-test, and ground-truth landmark
checks come from the TGOS Taichung CSV (which carries both coordinate
systems for every row — the data itself is the projection's reference).

## Verification

The verifier (`verify_samples.py`) is the single most important quality
gate in the pipeline. For each TGOS county we draw 50 anchor samples
from each of the four cardinal extremes (200 per county) and run six
checks per sample, totalling 2,400 assertions. See the manual for the
check list and tolerance budget.

## Output packaging

`build_manifest.py` ZIPs the sqlite files into four kits:

- `base.zip` — townships + roads + places-osm (must-have)
- `places-<county>.zip` — per-county TGOS sqlite (optional)
- `tw-central-full.zip` — everything in one bundle (convenience)

Each ZIP gets a sidecar `*.manifest.txt` carrying ZIP SHA-256, per-file
SHA-256, source CSV SHA-256, TGOS data date, OSM extraction counts, ISO
build timestamp, and region bbox. This mirrors the upstream VNS
generator's manifest discipline.
